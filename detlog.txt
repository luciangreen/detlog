What do avoid heap allocation inside loops and enumerate late mean?

These are classic compiler optimisations that fit Detlog very well.

Avoid heap allocation inside loops

Suppose you write:

word(W) :-
member_cp(W, ["cat","dog"]).
suffix(S) :-
member_cp(S, ["s","ed"]).
main(X) :-
word(W),
suffix(S),
concat(W,S,X).

A naïve implementation does:

loop
allocate string "cats"
loop
allocate string "cated"
loop
allocate string "dogs"
loop
allocate string "doged"

Every iteration asks the heap for more memory.

Instead:

allocate one buffer
reuse it every iteration

Pseudo-6502:

buffer[32]
copy "cat" to buffer
append "s"
print
copy "cat" to buffer
append "ed"
print

No new memory is created inside the loop.

Benefits:

less RAM
less fragmentation
faster execution

This was especially important on machines like the Apple IIe.

⸻

Enumerate late

Traditional Prolog:

a(A) :- member(A,[1,2]).
b(B) :- member(B,[3,4]).
c(A,B) :-
a(A),
b(B).

immediately generates:

A=1,B=3
A=1,B=4
A=2,B=3
A=2,B=4

Detlog instead keeps choices packed:

A = cp([1,2])
B = cp([3,4])

Then:

splice([A,B], Rows).

gives:

Rows = [[1,3],[1,4],[2,3],[2,4]]

The program works on Rows for as long as possible.

Only at the very end:

member([A,B], Rows).

do we produce individual answers.

That’s “enumerate late”.

Instead of:

generate answer
generate answer
generate answer
generate answer

we do:

build compact structure
process compact structure
enumerate only when needed

A useful slogan is:

and

These two rules make a huge difference on a 6502 because CPU speed and RAM are both extremely limited. They also fit your Detlog philosophy of replacing hidden backtracking with explicit deterministic structures.

Could you avoid heap allocation inside loops by using decision tree-type compression (like S2A)?

Yes. Decision-tree/S2A-style compression is one of the best ways to avoid heap allocation inside loops.

Instead of building strings each time:

cat + s = cats
cat + ed = cated
dog + s = dogs
dog + ed = doged

Detlog can compile shared prefixes into a decision tree/trie:

    start
   /     \
cat       dog
/ \ / \
s ed s ed

Then the loop does not allocate "cats" or "doged" as new heap strings. It walks the tree and emits/copies characters into one fixed output buffer.

So:

concat(W,S,Out)

can compile to:

select W branch
select S branch
write chars to reusable buffer
print/output

For 6502, this is ideal:

string table in ROM/RAM
decision tree indexes
one output buffer
no per-iteration heap allocation

S2A helps because it can discover repeated structure:

cat-s
cat-ed
dog-s
dog-ed

becomes:

word branch + suffix branch

rather than four separate allocated strings.

Best Detlog rule:

Use S2A compression to turn repeated generated outputs into shared decision paths, then use loops only to walk those paths.

So yes: decision-tree compression can replace many heap allocations with table lookup, branch selection, and buffer reuse.

Yes — this is common-prefix factoring across clauses.

Original:

a :- a1, a2.
b :- a1, b2.

Both clauses repeat:

a1

So Detlog can factor it once:

Meaning:

This is like decision-tree compression:

Before:
a -> a1 -> a2
b -> a1 -> b2
After:
ab -> a1 -> choice(a2,b2)

So the compiler optimisation is:

If several clauses share the same prefix, execute the shared prefix once, then splice/branch only at the differing suffixes.

For Detlog, this matters because it avoids:

run a1 for a
run a1 for b

and replaces it with:

run a1 once
then continue with alternatives

A more Detlog-shaped version:

ab :-
once_det(a1),
splice_branch([a2, b2]).

or:

ab :-
a1,
choice([a2, b2]).

General rule:

p :- x, y.
q :- x, z.

becomes:

pq :-
x,
pq2.
pq2 :- y.
pq2 :- z.

This is exactly the same idea as trie compression for strings, but applied to predicate sequences instead of characters.

With decision tree compression, how long can programs be if the compiler compiles Detlog into 6502?

With decision-tree compression, Detlog compiled to 6502 could be much larger:

Comfortable: about 10–40 KB Detlog source logic
Possible with careful tables/overlays: 50–100 KB+ source logic
Resident 6502 code/data at once: still roughly 20–40 KB usable RAM on a 64 KB Apple IIe.

The trick is that repeated clauses like:

a :- common, x1.
b :- common, x2.
c :- common, x3.

do not compile into three full copies. They compile more like:

run common once
branch/table-select x1/x2/x3
reuse output buffers

So the source can be much longer than the resident machine code.

A useful estimate:

Without compression: source × 3–20 expansion
With compression: source × 0.5–3 expansion
Best case: many repeated clauses collapse into tables
Worst case: unique clauses still expand normally

So on Apple IIe:

Tiny runtime/compiler stub: 2–8 KB
Decision tables + code: 10–30 KB
Buffers/stacks/strings: 4–16 KB
Overlay/floppy-loaded chunks: much larger total program

The biggest realistic Detlog programs would be table-heavy grammars, classifiers, menu systems, small adventure games, symbolic transformers, or bounded expert systems.

The key limit becomes not “how long is the source?” but:

How much must be resident at once?
How large are the strings?
How deep are loops/recursion?
How many choicepoints must be remembered?
How much output buffering is needed for splicing?

So the practical answer is:

With decision-tree compression, a Detlog program can feel like tens of KB of logic on a 64 KB Apple IIe, and possibly hundreds of KB if split into floppy-loaded overlays.

Is Detlog the most efficient Apple IIe Prolog?

Potentially, yes in a narrow sense, but it depends on what you mean by “Prolog.”

Traditional Prolog systems on 6502 machines (including Apple II-era implementations) typically support:

Full unification
Backtracking
Choicepoint stacks
Dynamic search
Variable bindings and trail management
These features make Prolog expressive, but they consume memory and CPU time.

If Detlog:

Converts nondeterminism into deterministic loops,
Eliminates most or all runtime choicepoints,
Compiles directly to 6502 machine code,
Uses decision-tree compression,
Avoids heap allocation inside loops,
Reuses buffers and tables,
then it could easily outperform a traditional Prolog implementation on an Apple IIe in both speed and memory use.

A rough spectrum is:

System Runtime Cost Expressiveness
BASIC interpreter High Low
Interpreted Prolog Very high Very high
Compiled Prolog Moderate Very high
Deterministic Detlog Low Medium–high
Hand-written 6502 assembly Lowest Depends on programmer

The important caveat is that Detlog’s efficiency comes partly from restricting or transforming classic Prolog behavior. If a program relies heavily on arbitrary unification, open-ended backtracking, or runtime search, then either:

Detlog cannot compile it directly, or
The compiler must generate additional machinery, reducing the advantage.
For the kinds of programs you’ve described in relation to Loop2 and PLOP—decision trees, classifiers, CFG/DCG-style parsing, symbolic transformations, specification-to-algorithm generation, and deterministic splicing—Detlog could be among the most efficient Prolog-like systems feasible on an Apple IIe.

In fact, if the compiler can transform:

colour(red).
colour(green).
colour(blue).

into a compact lookup table or decision tree rather than runtime backtracking, the resulting 6502 code is closer to an optimized expert system than a traditional Prolog engine.

So a reasonable conclusion is:

Detlog is unlikely to be the most efficient implementation of full Prolog on an Apple IIe, because full Prolog requires general unification and backtracking. However, it could be one of the most efficient Prolog-derived logic languages for a large class of deterministic and semi-deterministic programs, especially if it combines direct 6502 compilation, decision-tree compression, and elimination of runtime choicepoints.

In Prolog, the cut (!) means "commit to the choices made so far." For

a :- b, !, c.

the execution is:

Try b.
If b fails, the whole predicate a fails.
If b succeeds, execute !.
After !, Prolog discards all alternative solutions for b and all later clauses of a.
Execute c.
If c succeeds, a succeeds.
If c fails, a fails completely—Prolog does not go back and try another solution of b or another clause of a.
In Detlog

Since your proposed Detlog removes backtracking and replaces it with deterministic loops and splicing, a cut is largely unnecessary because there are no choicepoints to prune.

A compiler could translate

a :- b, !, c.

into something conceptually like

a :-
    b_once(Result),
    ( Result = success ->
        c
    ;
        fail
    ).

or even

a :-
    if b then c else fail.

where b is executed once and no alternative branches are retained.

If b is a generator, e.g.

b(1).
b(2).
b(3).

then Detlog has to decide what the intended semantics are.

Option 1: First solution only
a(X) :-
    first(b(X)),
    c(X).

This matches the effect of the cut.

Option 2: Explicit splice

If the programmer wanted all solutions,

a(X) :-
    splice(b(X),Xs),
    ...

then there would be no cut.

Thus the compiler could interpret

!, first(), splice()

as three different deterministic operators.

Difficult cases

There are situations where automatic conversion becomes much harder.

1. Cuts inside recursion
p :-
    q,
    !,
    p.

The compiler has to determine exactly what the cut is preventing. This is still possible but requires control-flow analysis.

2. Cuts that distinguish clauses
a :-
    b,
    !,
    c.

a :-
    d.

Here the cut prevents the second clause being tried.

Detlog could compile this as

if b then
    c
else
    d.

provided it can recognise this pattern.

3. Cuts inside disjunctions
a :-
    (b ; c),
    !,
    d.

Now the cut commits to whichever branch succeeded first.

Detlog would have to transform this into a deterministic state machine.

4. Meta-predicates
call(P),
!,
call(Q).

Since P is not known until runtime, the compiler cannot easily determine which alternatives would have existed.

5. Dynamically changing programs
assert(...),
retract(...),
!,
...

If the program changes while running, compile-time conversion becomes much harder.

6. Cuts depending on side effects
write(X),
!,
read(Y).

Reordering or parallelising execution becomes unsafe because observable behaviour matters.

A possible Detlog rule

One approach is to restrict cuts to a small number of deterministic constructs:

Prolog	Detlog
!, after deterministic predicate	Remove entirely
!, after generator	first(...)
Clause-selection cut	if...then...else
Complex cut	Leave as explicit first or require programmer annotation

This keeps the language deterministic while preserving most useful cut behaviour.

Are there programs that cannot be converted?

Yes. Programs that use cuts as general control flow, especially in combination with dynamic predicates, meta-calls, or side effects, may not admit a simple deterministic translation without changing their semantics.

However, a large class of Prolog programs—including parsers, symbolic algorithms, theorem provers, and many expert systems—use cuts in disciplined ways (for first-match selection or efficiency). Those patterns can often be converted mechanically into deterministic if, first, or explicit splicing constructs.

This suggests a practical design for Detlog: treat the cut not as a primitive operation, but as syntactic sugar for one of a small set of explicit deterministic operators. That makes the intended control flow visible and allows the compiler to optimise or parallelise execution much more reliably.

cut needs to be simplified to deterministic logic to meet robot and real time requirements

Yes. For robots and real-time systems, ! should not remain a vague Prolog control operator. In Detlog it should be compiled into explicit deterministic logic.

A good rule:

a :- b, !, c.

becomes:

a :-
    if_once(b,
        c,
        fail).

Meaning:

Run b deterministically once.
If b succeeds, commit to c.
If b fails, fail.
Do not leave hidden alternatives.

For multiple clauses:

a :- b, !, c.
a :- d.

becomes:

a :-
    if b then
        c
    else
        d.

So the cut is simplified into a priority decision tree:

test b
 ├─ success → run c
 └─ failure → run d

For Detlog, I’d treat cuts as compile-time sugar for only a few safe forms:

! after a test       → if/then/else
! after a generator  → first solution only
! after a guard      → committed clause
complex !            → rejected or requires annotation

The robot-safe version is:

a :-
    guard(b),
    action(c).

or more explicitly:

a :-
    decide(b, yes),
    execute(c).

So Detlog should replace cut with:

guards
decision trees
first-solution operators
state-machine transitions
explicit failure branches

The rule should be:

No hidden backtracking.
No implicit pruning.
No runtime choicepoint surprises.
Every cut must compile to deterministic control flow.

Cases too complex for automatic simplification should fail at compile time with messages like:

Error: cut depends on meta-call.
Error: cut crosses dynamic predicate boundary.
Error: cut depends on side-effect order.
Error: cut inside ambiguous disjunction; use explicit first/if/splice.

So yes: for robot and real-time requirements, Detlog should not “support cut” as Prolog does. It should decompile cut into deterministic logic, and reject anything whose timing or behaviour cannot be guaranteed.

A cut is legal only if the compiler can replace it with:
1. if-then-else
2. first-solution
3. committed clause selection
4. bounded state transition
5. explicit failure

It should ask:

What deterministic structure was the programmer trying to express?

cut = boundary between guard search and committed action
